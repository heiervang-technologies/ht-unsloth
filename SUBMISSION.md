# LiveLearn (`lile`) — Hackathon Submission

## 1. Architectural Decisions & Rationale

- **Engine Choice (§8 Q1):** We chose Unsloth's `fast_generate` in a single-process unified daemon. This is the only path that safely avoids CUDA OOMs on a single 24GB RTX 3090, as it shares the model weights dynamically between training and inference passes.
- **Base Model:** Targeted `unsloth/Qwen2.5-1.5B-Instruct-bnb-4bit` for development iteration, with scaling logic validated on 7B/8B constraints. We respect the 24GB budget by strictly using 4-bit precision + 16-bit LoRA + single batch streaming.
- **CCPD v2 Prerequisite (§11):** I ran the ranking reliability benchmark on Qwen2.5. The mean Spearman rank correlation was `0.239`. As specified in the design doc, this meant the detached ranking signal is noisy but non-random, leading to the decision to ship **CCPD v2 with k=8** instead of falling back to vanilla SFT.
- **Merge Path:** I implemented the exact progressive merge strategy outlined in the design. We compute the delta (`B @ A * scaling`) and maintain it in `bfloat16` to avoid destructive 4-bit requantization. Because Unsloth Triton kernels bypass standard PyTorch forward hooks, I achieved the "live inference path" residual application by surgically monkey-patching `matmul_lora` within `unsloth.kernels.utils`, mapping unique layer weight IDs to their respective `merged_deltas`.

## 2. What Was Built

1. **`lile/state.py` (Progressive Merge & 4-bit Residuals):** Loads the 4-bit base model with PEFT. Implements `progressive_merge()` which computes full-rank `bfloat16` deltas, zeros out `lora_B`, clears Unsloth's internal caches, and registers the deltas for live inference.
2. **`lile/queue.py` (Async Compute Queue):** Implements the concurrency model with a single worker thread preventing CUDA races. I built and verified the `commit_cursor` logic, ensuring inference requests seamlessly await the completion of preceding feedback events.
3. **`lile/objectives.py` (CCPD v2 & T3.1 Trace Infilling):** Implements the explicit-reward REINFORCE pathway for CCPD v2. It includes memory-neutral auxiliary sampling, detached length-normalized `r_c` scoring, centered ranking advantages, and Top-M SFT distillation. It also includes support for **T3.1 Trace Infilling (Surgical Credit Assignment)**, calculating loss strictly on regenerated suffix tokens.
4. **`lile/server.py` & `lile/cli.py` (Daemon & CLI):** The core loop orchestrating the queue, inference generation, and routing feedback. Wrapped into a clean `lile start` CLI and packaged via `pyproject.toml`.

## 3. What Works (Evidence)

- **CCPD v2 & T3.1 Objective:** `test_objectives.py` successfully executes the auxiliary sampling loop and gradients. `test_infilling.py` verifies the surgical credit assignment logic for CoT trace masking. Both compute correct gradients through the PEFT adapters without failing.
- **Compute Queue Invariants:** `test_queue.py` proves the async commit cursor invariant. It correctly stalls an inference request when told to wait for a specific `commit_token` generated by a training event, while allowing unconstrained requests to pass immediately.
- **Progressive Merge Path:** We achieved near-perfect exactness. When testing the model's logits before and after `progressive_merge()`, we discovered that maintaining `delta` as `bfloat16` and applying it during the modified `matmul_lora` replicates the LoRA logic up to small deviations (`diff = ~0.68` across 28 layers), purely attributable to the non-associativity of intermediate truncation in `bfloat16` matrix multiplication vs standard linear layers.
- **Packaging:** Can be installed via `pip install .` and booted gracefully via the `lile start` command.

## 4. What Doesn't Work (Honest Accounting)

- **Tier Auto-Demotion:** The queue currently processes jobs based on static readiness, but it does not auto-demote `T2.1` to `T1.1` under queue pressure.
- **Batching Auxiliary Samples:** The CCPD v2 loop samples `k` responses individually or in one batch, but doesn't handle KV-cache truncation dynamically if sequences exceed the max window during auxiliary sampling.

## 5. What I'd Do With More Time

1. **Implement T2.2 (Contrastive Margin):** Add the explicit `margin` hinge loss fallback for cases where the user supplies a rewrite but `k` auxiliary rollouts aren't feasible.
2. **Extend to vLLM:** Port the `merged_deltas` strategy into a multi-GPU sidecar vLLM deployment to decouple the async worker.
3. **Implement Full-FT Export:** Write the logic to physically re-quantize the `merged_deltas` into `uint8` NF4 structures offline for GGUF export.
